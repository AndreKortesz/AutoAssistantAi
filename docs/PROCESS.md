# Процесс сборки данных по модели — AutoAssistantAi

> Версия 4.0 (с независимыми моделями для аудита, учётом близнецов, feedback loop)
> Обновлено: 30.04.2026

## Зачем нужен этот процесс

Каждое поколение автомобиля в БД AutoAssistantAi проходит **10 шагов** перед публикацией и поддерживается через регулярный аудит после.

В версии 4.0 учтены 6 структурных пробелов из v3.0:
1. **Claude проверял Claude** на разных шагах — это иллюзия независимости. Теперь аудит обязательно делает Grok или Gemini.
2. **Промпты не учитывали близнецов** модели в других странах — теперь явно требуется собирать данные с зарубежных аналогов (Hyundai Solaris RB = Accent RB США = Verna RB Индия).
3. **Не было процесса поддержки** после публикации — добавлены шаги 9 (feedback loop) и 10 (quarterly audit).
4. **Не накапливался опыт отбраковки** — теперь ведётся `discarded_global.json`.
5. **Web-search ≠ OEM-каталог** — введён whitelist «опасных полей» требующих ручной проверки разработчиком.
6. **Время сборки было занижено** — реальные цифры: 9-11 часов на поколение вручную.

## История критических ошибок

**Solaris RB (26.04.2026):** артикул `04500-00115` был указан как «Hyundai ATF SP-III» для коробки A4CF1, но это **SP-IV для 6-ст коробок A6GF1**. Заливка SP-IV в A4CF1 необратимо повредила бы коробку у пользователя. Без аудита эта ошибка попала бы в production. **Аудит обязателен.**

## Полный процесс (10 шагов)

### Шаг 0. Подготовка — определить близнецов модели за границей

**Это новый обязательный шаг в v4.0.** Многие модели РФ-рынка имеют конструктивно идентичных близнецов в других странах:

| Модель в РФ | Близнец в США | Близнец в Азии/ЕС |
|--------------|----------------|---------------------|
| Hyundai Solaris RB | Hyundai Accent RB | Hyundai Verna RB (Индия) |
| Kia Rio III QB | Kia Rio (USA) | — |
| VW Polo Sedan Mk5 (RU) | — | VW Vento (Индия) |
| Renault Logan II | Dacia Logan II (ЕС) | — |
| Skoda Rapid (RU) | — | Skoda Rapid (Индия) |

**Зачем это нужно:** recalls и class actions документированы в США (NHTSA), Канаде (Transport Canada), ЕС (KBA Германии) **гораздо лучше** чем в РФ. Если не указать близнецов — данные о реальных дефектах с подтверждением будут пропущены.

**На примере Solaris RB:**
- В РФ есть только 1 recall (Росстандарт 2025, HECU)
- В США как Accent RB есть **4 recalls** (NHTSA 23V-651, 15V-566, 15V-353, 20V-393)
- Все 4 recalls конструктивно применимы к Solaris РФ (тот же VIN-диапазон электроники)

**Перед запуском промптов 1-5 — заполнить шаблон:**

```
МАРКА: Hyundai
МОДЕЛЬ: Solaris
ПОКОЛЕНИЕ: I (RB)
ГОДЫ: 2010-2017
ДВИГАТЕЛИ: G4FA 1.4 (107 л.с.) MPI, G4FC 1.6 (123 л.с.) MPI
КОРОБКИ: M5CF1 (МКПП 5), M6CF1 (МКПП 6, рест), A4CF1 (АКПП 4), A6GF1 (АКПП 6, рест)

БЛИЗНЕЦЫ (учитывать при сборе):
- США: Hyundai Accent RB (2012-2017)
- Индия: Hyundai Verna RB (2010-2017)
- Канада: Hyundai Accent RB (2012-2017)
```

Этот блок подставляется в начало **каждого** промпта (1-5). Конкретно в шаге 4 (recalls/lawsuits) — обязательно искать данные в NHTSA / Transport Canada / KBA по близнецам.

### Шаг 1. Claude × 5 промптов — глубокое исследование

Запустить в Claude.ai (отдельный чат) промпты 1-5 поочерёдно:

| Промпт | Задача |
|--------|--------|
| `prompt-1-engine.txt` | Двигатель: ГРМ, турбина, масложор, задиры, форсунки, прокладки |
| `prompt-2-transmission.txt` | Коробки передач: гидроблок, сцепление, мехатроник, подшипники |
| `prompt-3-other-systems.txt` | Подвеска, тормоза, рулевое, электрика, охлаждение, кузов |
| `prompt-4-recalls-lawsuits.txt` | Recalls, class actions, TSB по всем странам |
| `prompt-5-wear-maintenance.txt` | Износ (амортизаторы, колодки, ШРУС) и ТО (масла, фильтры) |

**Подставить:** `[МАРКА] [МОДЕЛЬ] [ПОКОЛЕНИЕ] ([ГОДЫ])` в начале каждого промпта.

**Результат:** 5 файлов JSON от Claude, по одному на каждый блок.

### Шаг 2. Grok × 5 промптов — параллельное исследование

В Grok запустить **те же** 5 промптов с теми же подстановками. Можно параллельно с Шагом 1.

**Результат:** 5 файлов JSON от Grok.

**Зачем дубликат:** перекрёстная проверка. Один LLM пропускает то, что другой замечает. На Solaris RB Grok правильно нашёл слетающую крыльчатку вентилятора, но Claude его пропустил — и наоборот.

### Шаг 3. Перекрёстная проверка Claude ↔ Grok

В этом же чате Claude (где собирали данные на Шаге 1) для каждого из 5 блоков:

1. Скопировать ответ Grok
2. Вставить в чат Claude с просьбой: «Проверь эти данные от Grok, найди ошибки и галлюцинации»
3. Получить от Claude **готовый текст-запрос для Grok** с найденными ошибками
4. Вставить этот запрос в Grok → получить исправленный JSON
5. Сохранить исправленный JSON

**Результат:** 5 проверенных файлов Claude + 5 проверенных файлов Grok = **10 файлов**.

### Шаг 4. Промпт 6 — сводка из 10 файлов

В Grok загрузить все 10 файлов и применить `prompt-6-summary.txt`. Получить сводку (top-5, рейтинг двигателей и коробок, чеклист, бюджет).

Затем в Claude.ai загрузить **ту же сводку Grok** + 10 исходных файлов и попросить Claude **проверить сводку Grok**. Claude найдёт ошибки в сводке (как в нашем `grok-summary-review.md`).

**Результат:** проверенная и исправленная сводка.

### Шаг 5. Финальная сборка JSON в Claude.ai (с встроенной проверкой)

В **новом** чате Claude (чтобы был чистый контекст):
1. Загрузить все 10 файлов + проверенную сводку
2. Загрузить файлы `issue-gold-standard-schema.json` и `issue-example-mustang-fuel-sensor.json` для контекста схемы
3. Попросить Claude собрать финальный гибридный JSON по схеме v1.1

**ВАЖНО: Сборка состоит из 5 подшагов. Все они должны быть выполнены — иначе аудит на шаге 6 найдёт ошибки которые должны были быть отловлены раньше.**

#### 5.1. Слияние 10 файлов
Объединить все записи Claude и Grok в единый набор. Удалить очевидные дубликаты (одна и та же болячка в обоих источниках).

#### 5.2. Поиск конфликтов Claude vs Grok
Для каждой записи где данные расходятся — зафиксировать конфликт. Например:
- Claude: `frequency_percent: 35`, Grok: `frequency_percent: 60`
- Claude: «recall в 2018», Grok: «recall в 2019»
- Claude: `part_number: BU5Z-9F972-B`, Grok: `part_number: BU5A-9F972-CA`

Эти расхождения **не игнорировать** и не выбирать по принципу «кто первый написал».

#### 5.3. ⚠️ Web-search верификация (КРИТИЧЕСКИ ВАЖНО)
Это новый шаг в версии 3.0. Сейчас он защищает от галлюцинаций до того как они попадут в JSON.

Для каждого спорного / критичного значения — проверить через web-search:
- **Все `recall_id` и `campaign_code`** — открыть NHTSA / Росстандарт / Transport Canada / KBA и подтвердить
- **Все `case_number`** — проверить в правовых базах (PACER, court listener)
- **Все `part_number`** для systemic_defect — найти в OEM-каталогах (HyundaiPartsDeal, ETKA, EPC, hyundaipartsdeal.com)
- **Все конфликты Claude vs Grok** — определить кто прав через первоисточник
- **Все `affected_units`** — подтвердить количество в источнике

Если данные не подтверждаются — **поставить `null` + `meta.needs_review: true`**. Не угадывать.

#### 5.4. Дедупликация и нормализация
- Удалить полные дубли
- Слить частичные дубли (одна болячка с разными деталями) в одну запись с обогащёнными полями
- Привести типы данных к схеме (массивы → массивы, числа → числа)

#### 5.5. Запись предварительного JSON
Сохранить результат как `{model}_final_pre_audit.json`. Это вход для шага 6.

**Результат:** `{model}_final_pre_audit.json` — JSON в формате схемы v1.1, **с уже проверенными артикулами и recall_id**.

### Шаг 6. АУДИТ через prompt-7-audit (Grok или Gemini, СТРОГО НЕ Claude!)

**Это критически важный шаг — не пропускать.**

**ВАЖНО в v4.0:** аудит делается **другой моделью**, не Claude. Причина — Claude в шаге 5 и Claude в шаге 6 это **один LLM в разных чатах**. Если у Claude в обучении нет данных по конкретному артикулу — он не узнает это в любом из 5 чатов. Меняется только контекст разговора, но не знания модели.

Реальную независимость даёт **смена модели**, не смена чата.

#### 6.1. Какую модель использовать для аудита

| Модель | Когда использовать | Цена |
|--------|---------------------|------|
| **Grok** | Default — у тебя уже есть подписка | $0 |
| **Gemini** | Если Grok недоступен или нужна альтернатива | $0 (бесплатный план) |
| **GPT-4** | Если есть подписка ChatGPT | $0 (по подписке) |
| ❌ Claude | **НЕ использовать** для аудита | — |

#### 6.2. Как запустить

В Grok (или выбранной модели):
1. Загрузить готовый `{model}_final_pre_audit.json` (результат шага 5)
2. Применить `prompt-7-audit.txt` с подстановкой марки/модели/поколения **и блока близнецов из шага 0**

#### 6.3. Что находит аудитор

- **Галлюцинации артикулов** (несуществующие part_number)
- **Пропущенные болячки** с подтверждениями из источников
- **Пропущенные recalls / class actions / TSB** (особенно по близнецам)
- **Несостыковки** (даты, статусы, формулировки)
- **Структурные ошибки** (broken `linked_issue_id`, типы полей)

#### 6.4. Чего аудитор НЕ делает (важно!)

- ❌ **НЕ меняет JSON** — только пишет отчёт
- ❌ **НЕ добавляет новые записи** в файл
- ❌ **НЕ применяет исправления** автоматически

Это защитная архитектура. Если бы аудитор сразу мог менять JSON — он бы менял его на основе своих галлюцинаций тоже. Разделение нужно чтобы каждый шаг был проверяем независимо.

**Результат:** структурированный отчёт по разделам А-Е (галлюцинации / пропуски / recalls / несостыковки / структурные ошибки / сводка).

### Шаг 7. Применение правок в Claude.ai (с двойной защитой и whitelist опасных полей)

Вернуться в основной чат сборки (или создать новый). Передать Claude отчёт аудита.

**ВАЖНО: применение правок состоит из 4 подшагов. Подшаги 7.1 и 7.2 — критичны, не пропускать.**

#### 7.1. ⚠️ Повторная проверка источников аудитора

Перед тем как добавить новую болячку или исправить артикул на основе отчёта аудитора — **самостоятельно проверить через web-search**:

- Если аудитор сказал «правильный артикул 56521-4L000» — открыть каталог и подтвердить
- Если аудитор сказал «пропущена болячка X с источниками 1, 2, 3» — открыть эти источники и убедиться что они реальные и говорят то что сказал аудитор
- Если аудитор сказал «recall 15V-566 пропущен» — найти его в NHTSA

**Зачем это нужно:** аудитор тоже LLM, тоже может галлюцинировать. Без 7.1 мы доверяем одному источнику (отчёту аудитора) без перепроверки.

**Что делать с непроверяемыми правками:**
- Если источник аудитора реален → применить правку
- Если источник аудитора не открывается / содержит другую информацию → **пропустить правку**, отметить в `MERGE_REPORT.md` как «не подтверждено»
- Если есть сомнение → пометить запись `meta.needs_review: true`

#### 7.2. ⚠️ Whitelist «опасных полей» — ручная проверка разработчиком

**Это новый шаг в v4.0.** Web-search ≠ доступ к OEM-каталогам. LLM не может зайти в Hyundai Mobis Web EPC, Kia Microcat, ETKA — там требуется логин и подписка. Web-search возвращает сайты второго уровня (Exist, Autodoc, разборки), которые сами могут содержать ошибки.

Поэтому для **полей с критическим production-impact** требуется ручная проверка человеком. Список «опасных полей» хранится в `data/critical_fields.json`:

```json
{
  "critical_fields_v1": [
    {
      "field": "consumables.part_number",
      "trigger": "name contains ATF/ATF SP-III/ATF SP-IV/трансмиссионное",
      "reason": "Заливка неправильного ATF необратимо повреждает АКПП",
      "verification_required": "manual_oem_catalog"
    },
    {
      "field": "consumables.part_number",
      "trigger": "name contains тормозная жидкость/DOT",
      "reason": "Несовместимая жидкость → отказ тормозов",
      "verification_required": "manual_oem_catalog"
    },
    {
      "field": "consumables.part_number",
      "trigger": "name contains антифриз/coolant",
      "reason": "Несовместимый антифриз → коррозия двигателя",
      "verification_required": "manual_oem_catalog"
    },
    {
      "field": "parts.part_number",
      "trigger": "issue.subsystem contains airbag/srs",
      "reason": "Неправильная подушка безопасности → не сработает в ДТП",
      "verification_required": "manual_oem_catalog"
    },
    {
      "field": "parts.part_number",
      "trigger": "issue.subsystem contains brake_caliper/brake_pad",
      "reason": "Неподходящие тормозные колодки → отказ тормозов",
      "verification_required": "manual_oem_catalog"
    }
  ]
}
```

**Процедура для опасных полей:**
1. После шага 7.1 — Claude помечает все «опасные поля» в JSON флагом `manual_check_required: true`
2. Перед production — разработчик проверяет каждое такое поле в реальном OEM-каталоге
3. После проверки → флаг меняется на `manual_check_passed: true` с датой и инициалами

**До покупки OEM API** (см. дорожную карту) этот список — **единственная защита** от ATF-SP-III/SP-IV-подобных инцидентов.

#### 7.3. Применение проверенных правок

Только после 7.1 и 7.2 для каждой подтверждённой ошибки:
- **Исправить артикулы** в существующих записях
- **Добавить пропущенные болячки** как новые `systemic_defect` (с полным набором полей по схеме v1.1)
- **Добавить пропущенные recalls** в `global_recalls`
- **Исправить broken ссылки** (`linked_issue_id`)
- **Исправить структурные ошибки** (типы полей, формулировки)

#### 7.4. Обновление сопутствующих файлов

- **`{model}_buyer_guide.md`** — добавить новые болячки в чеклист, обновить рейтинги
- **`MERGE_REPORT.md`** — добавить секцию «Аудит данных» с описанием:
  - Применённых правок
  - Отклонённых правок (на 7.1) с причиной
  - Полей помеченных `manual_check_required: true` (на 7.2)
- **`data/discarded_global.json`** — добавить новые отброшенные болячки (см. шаг 7.5)

#### 7.5. Обновление глобального реестра отброшенных болячек

**Это новый шаг в v4.0.** При сборке каждого нового поколения накапливается опыт «эту болячку мы уже отбросили один раз — почему отбрасывать снова?». Чтобы не дублировать работу, ведём `data/discarded_global.json`:

```json
{
  "version": "1.0",
  "updated_at": "2026-04-30",
  "entries": [
    {
      "canonical_issue": "thermostat_sticking",
      "brands_discarded": ["hyundai_solaris_rb", "kia_rio_qb"],
      "reason": "Единичный износ после 100к, не специфика конкретной модели",
      "decision_rule": "Если для нового поколения <3 источников и средний пробег > 100к — отбрасывать без расследования"
    },
    {
      "canonical_issue": "door_seals_freeze",
      "brands_discarded": ["all"],
      "reason": "Общая проблема всех авто на холодном пуске, не специфика модели",
      "decision_rule": "Никогда не включать в systemic"
    }
  ]
}
```

**Использование:** при запуске сборки нового поколения — выгрузка релевантных записей из реестра подаётся Claude в контекст. Claude видит «вот что мы отбрасывали раньше, не повторяй ошибки».

**Результат:** обновлённые 4 файла:
- `{model}_final.json` (исправленный, версия после аудита)
- `{model}_buyer_guide.md` (с новыми болячками)
- `MERGE_REPORT.md` (с секцией про аудит)
- `data/discarded_global.json` (с новыми записями отброшенных)

### Шаг 8. Импорт в БД AutoAssistantAi

Положить файлы в проект Railway:

```
data/
└── issues/
    └── {brand}/                    ← hyundai, kia, toyota, ...
        └── {generation_slug}/      ← solaris_rb, rio_qb, camry_xv70, ...
            ├── {model}_final.json
            ├── {model}_buyer_guide.md
            └── MERGE_REPORT.md
```

После импорта данные показываются в UI на трёх страницах:

- **Здоровье авто** — `common_wear` (по пробегу)
- **ТО** — `maintenance` (по интервалам)

### Шаг 9. Feedback loop — обратная связь от пользователей

**Это новый шаг в v4.0.** После публикации данных в production пользователи приложения **обнаруживают ошибки** которые не были найдены аудитом:
- Артикул не подходит к их машине
- Цена не соответствует реальности
- Решение не сработало
- Болячка проявилась раньше/позже чем указано

Без механизма сбора этой обратной связи ошибки **накапливаются молча** и снижают доверие к приложению.

#### 9.1. UI-механизм обратной связи

На каждой странице записи в приложении — кнопка «Сообщить об ошибке». При нажатии открывается форма:
- **Тип ошибки:** артикул не подходит / цена устарела / решение не сработало / другое
- **Что не так:** свободный текст
- **Подтверждение:** опционально — фото / ссылка / номер VIN

Отчёты копятся в БД-таблице `user_feedback` со связью на `record_id` и `field_name`.

#### 9.2. Триаж отчётов (ежемесячно)

Раз в месяц разработчик / аналитик проходит накопленные отчёты:
- **3+ отчётов на одну запись с одним типом ошибки** → `meta.needs_review: true`, проверка вручную
- **5+ отчётов** → запись блокируется в UI до проверки
- **1-2 отчёта** → отметка для следующего quarterly audit

---

### Шаг 10. Quarterly audit — регулярный аудит топ-моделей

**Это новый шаг в v4.0.** Данные устаревают. Цены меняются за 6 месяцев. Recalls добавляются производителями. Появляются новые жалобы Drive2/Drom за прошедший период.

Без регулярного обновления через год работы данные становятся неточными по 30-50% полей.

#### 10.1. Что аудитим раз в квартал

**Топ-20 моделей по трафику в приложении** + все модели с накопленным feedback:
- Цены (`price.as_of_date` старше 6 месяцев → перепроверить)
- Recalls (новые отзывные кампании за квартал)
- Class actions (статусы могли измениться: pending → settlement)
- Жалобы Drive2/Drom за последние 3 месяца

#### 10.2. Процедура

Для каждой модели из списка:
1. Запустить **prompt-7-audit** через Grok заново
2. Собрать накопленный пользовательский feedback (из шага 9)
3. Применить правки по шагу 7 (с защитами 7.1 и 7.2)
4. Обновить `version` в JSON и `updated_at` дату
5. Пуш в production

#### 10.3. Метрики качества

После каждого quarterly audit фиксировать:
- Количество найденных устаревших цен
- Количество новых recalls
- Количество исправленных артикулов
- Количество feedback-отчётов закрыто

Эти метрики дают понимание **насколько данные стареют** между аудитами и нужно ли увеличить частоту.

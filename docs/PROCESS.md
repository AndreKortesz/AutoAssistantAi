# Процесс сборки данных по модели — AutoAssistantAi

> Версия 3.0 (с двойной защитой от галлюцинаций)
> Обновлено: 30.04.2026

## Зачем нужен этот процесс

Каждое поколение автомобиля в БД AutoAssistantAi должно проходить **8 шагов** перед публикацией. Шаги 6-7 (аудит) — добавлены после опыта с Hyundai Solaris RB, когда обнаружились критические галлюцинации артикулов, которые без аудита попали бы в production.

Самый яркий пример опасной галлюцинации: артикул `04500-00115` был указан как «Hyundai ATF SP-III» для коробки A4CF1, но это **SP-IV для 6-ст коробок A6GF1**. Заливка SP-IV в A4CF1 необратимо повредила бы коробку у пользователя. **Без аудита эта ошибка попала бы в production.**

В версии 3.0 добавлены явные подшаги web-search верификации внутри сборки (5.3) и перед применением правок аудита (7.1). Это защита от ситуаций когда **аудит сам ошибается** или **сборщик доверяет данным без проверки**.

## Полный процесс (8 шагов)

### Шаг 1. Claude × 5 промптов — глубокое исследование

Запустить в Claude.ai (отдельный чат) промпты 1-5 поочерёдно:

| Промпт | Задача |
|--------|--------|
| `prompt-1-engine.txt` | Двигатель: ГРМ, турбина, масложор, задиры, форсунки, прокладки |
| `prompt-2-transmission.txt` | Коробки передач: гидроблок, сцепление, мехатроник, подшипники |
| `prompt-3-other-systems.txt` | Подвеска, тормоза, рулевое, электрика, охлаждение, кузов |
| `prompt-4-recalls-lawsuits.txt` | Recalls, class actions, TSB по всем странам |
| `prompt-5-wear-maintenance.txt` | Износ (амортизаторы, колодки, ШРУС) и ТО (масла, фильтры) |

**Подставить:** `[МАРКА] [МОДЕЛЬ] [ПОКОЛЕНИЕ] ([ГОДЫ])` в начале каждого промпта.

**Результат:** 5 файлов JSON от Claude, по одному на каждый блок.

### Шаг 2. Grok × 5 промптов — параллельное исследование

В Grok запустить **те же** 5 промптов с теми же подстановками. Можно параллельно с Шагом 1.

**Результат:** 5 файлов JSON от Grok.

**Зачем дубликат:** перекрёстная проверка. Один LLM пропускает то, что другой замечает. На Solaris RB Grok правильно нашёл слетающую крыльчатку вентилятора, но Claude его пропустил — и наоборот.

### Шаг 3. Перекрёстная проверка Claude ↔ Grok

В этом же чате Claude (где собирали данные на Шаге 1) для каждого из 5 блоков:

1. Скопировать ответ Grok
2. Вставить в чат Claude с просьбой: «Проверь эти данные от Grok, найди ошибки и галлюцинации»
3. Получить от Claude **готовый текст-запрос для Grok** с найденными ошибками
4. Вставить этот запрос в Grok → получить исправленный JSON
5. Сохранить исправленный JSON

**Результат:** 5 проверенных файлов Claude + 5 проверенных файлов Grok = **10 файлов**.

### Шаг 4. Промпт 6 — сводка из 10 файлов

В Grok загрузить все 10 файлов и применить `prompt-6-summary.txt`. Получить сводку (top-5, рейтинг двигателей и коробок, чеклист, бюджет).

Затем в Claude.ai загрузить **ту же сводку Grok** + 10 исходных файлов и попросить Claude **проверить сводку Grok**. Claude найдёт ошибки в сводке (как в нашем `grok-summary-review.md`).

**Результат:** проверенная и исправленная сводка.

### Шаг 5. Финальная сборка JSON в Claude.ai (с встроенной проверкой)

В **новом** чате Claude (чтобы был чистый контекст):
1. Загрузить все 10 файлов + проверенную сводку
2. Загрузить файлы `issue-gold-standard-schema.json` и `issue-example-mustang-fuel-sensor.json` для контекста схемы
3. Попросить Claude собрать финальный гибридный JSON по схеме v1.1

**ВАЖНО: Сборка состоит из 5 подшагов. Все они должны быть выполнены — иначе аудит на шаге 6 найдёт ошибки которые должны были быть отловлены раньше.**

#### 5.1. Слияние 10 файлов
Объединить все записи Claude и Grok в единый набор. Удалить очевидные дубликаты (одна и та же болячка в обоих источниках).

#### 5.2. Поиск конфликтов Claude vs Grok
Для каждой записи где данные расходятся — зафиксировать конфликт. Например:
- Claude: `frequency_percent: 35`, Grok: `frequency_percent: 60`
- Claude: «recall в 2018», Grok: «recall в 2019»
- Claude: `part_number: BU5Z-9F972-B`, Grok: `part_number: BU5A-9F972-CA`

Эти расхождения **не игнорировать** и не выбирать по принципу «кто первый написал».

#### 5.3. ⚠️ Web-search верификация (КРИТИЧЕСКИ ВАЖНО)
Это новый шаг в версии 3.0. Сейчас он защищает от галлюцинаций до того как они попадут в JSON.

Для каждого спорного / критичного значения — проверить через web-search:
- **Все `recall_id` и `campaign_code`** — открыть NHTSA / Росстандарт / Transport Canada / KBA и подтвердить
- **Все `case_number`** — проверить в правовых базах (PACER, court listener)
- **Все `part_number`** для systemic_defect — найти в OEM-каталогах (HyundaiPartsDeal, ETKA, EPC, hyundaipartsdeal.com)
- **Все конфликты Claude vs Grok** — определить кто прав через первоисточник
- **Все `affected_units`** — подтвердить количество в источнике

Если данные не подтверждаются — **поставить `null` + `meta.needs_review: true`**. Не угадывать.

#### 5.4. Дедупликация и нормализация
- Удалить полные дубли
- Слить частичные дубли (одна болячка с разными деталями) в одну запись с обогащёнными полями
- Привести типы данных к схеме (массивы → массивы, числа → числа)

#### 5.5. Запись предварительного JSON
Сохранить результат как `{model}_final_pre_audit.json`. Это вход для шага 6.

**Результат:** `{model}_final_pre_audit.json` — JSON в формате схемы v1.1, **с уже проверенными артикулами и recall_id**.

### Шаг 6. АУДИТ через prompt-7-audit (отдельный чат, СТРОГО ПО ПРАВИЛАМ)

**Это критически важный шаг — не пропускать.**

В **отдельном** чате Claude (не связанном с проектом, чтобы не было влияния):
1. Загрузить готовый `{model}_final_pre_audit.json` (результат шага 5)
2. Применить `prompt-7-audit.txt` с подстановкой марки/модели/поколения

#### 6.1. Что делает аудитор

Находит:
- **Галлюцинации артикулов** (несуществующие part_number)
- **Пропущенные болячки** с подтверждениями из источников
- **Пропущенные recalls / class actions / TSB**
- **Несостыковки** (даты, статусы, формулировки)
- **Структурные ошибки** (broken `linked_issue_id`, типы полей)

#### 6.2. Чего аудитор НЕ делает (важно!)

- ❌ **НЕ меняет JSON** — только пишет отчёт
- ❌ **НЕ добавляет новые записи** в файл
- ❌ **НЕ применяет исправления** автоматически

Это защитная архитектура. Если бы аудитор сразу мог менять JSON — он бы менял его на основе своих галлюцинаций тоже. Разделение нужно чтобы каждый шаг был проверяем независимо.

**Результат:** структурированный отчёт по разделам А-Е (галлюцинации / пропуски / recalls / несостыковки / структурные ошибки / сводка).

### Шаг 7. Применение правок в Claude.ai (с двойной защитой)

Вернуться в основной чат сборки (или создать новый). Передать Claude отчёт аудита.

**ВАЖНО: применение правок состоит из 3 подшагов. Подшаг 7.1 — критичен, не пропускать.**

#### 7.1. ⚠️ Повторная проверка источников аудитора (новый шаг в v3.0)

Перед тем как добавить новую болячку или исправить артикул на основе отчёта аудитора — **самостоятельно проверить через web-search**:

- Если аудитор сказал «правильный артикул 56521-4L000» — открыть каталог и подтвердить
- Если аудитор сказал «пропущена болячка X с источниками 1, 2, 3» — открыть эти источники и убедиться что они реальные и говорят то что сказал аудитор
- Если аудитор сказал «recall 15V-566 пропущен» — найти его в NHTSA

**Зачем это нужно:** аудитор тоже LLM, тоже может галлюцинировать. Без 7.1 мы доверяем одному источнику (отчёту аудитора) без перепроверки.

**Что делать с непроверяемыми правками:**
- Если источник аудитора реален → применить правку
- Если источник аудитора не открывается / содержит другую информацию → **пропустить правку**, отметить в `MERGE_REPORT.md` как «не подтверждено»
- Если есть сомнение → пометить запись `meta.needs_review: true`

#### 7.2. Применение проверенных правок

Только после 7.1 для каждой подтверждённой ошибки:
- **Исправить артикулы** в существующих записях
- **Добавить пропущенные болячки** как новые `systemic_defect` (с полным набором полей по схеме v1.1)
- **Добавить пропущенные recalls** в `global_recalls`
- **Исправить broken ссылки** (`linked_issue_id`)
- **Исправить структурные ошибки** (типы полей, формулировки)

#### 7.3. Обновление сопутствующих файлов

- **`{model}_buyer_guide.md`** — добавить новые болячки в чеклист, обновить рейтинги
- **`MERGE_REPORT.md`** — добавить секцию «Аудит данных» с описанием всех применённых правок и тех что были отклонены на 7.1

**Результат:** обновлённые 3 файла:
- `{model}_final.json` (исправленный, версия после аудита)
- `{model}_buyer_guide.md` (с новыми болячками)
- `MERGE_REPORT.md` (с секцией про аудит)

### Шаг 8. Импорт в БД AutoAssistantAi

Положить файлы в проект Railway:

```
data/
└── issues/
    └── {brand}/                    ← hyundai, kia, toyota, ...
        └── {generation_slug}/      ← solaris_rb, rio_qb, camry_xv70, ...
            ├── {model}_final.json
            ├── {model}_buyer_guide.md
            └── MERGE_REPORT.md
```

После импорта данные показываются в UI на трёх страницах:
- **Болячки** — `systemic_defect` (с фильтром по двигателю/коробке)
- **Здоровье авто** — `common_wear` (по пробегу)
- **ТО** — `maintenance` (по интервалам)

## Стоимость и время

| Способ | Время на 1 поколение | Стоимость |
|--------|----------------------|-----------|
| Полностью вручную (с защитами 5.3 и 7.1) | 7-9 часов | $0 (по подписке Claude.ai + Grok) |
| Гибрид API + ручная сборка в Claude.ai | 1.5-2.5 часа | ~$14-20 за поколение |
| Полностью через API (n8n workflow) | 30-60 минут | ~$30 за поколение |

На все 518 приоритетных поколений:
- Полностью вручную — ~3.5-4.5 года работы
- Гибрид — ~$10 500 + 800-1300 часов работы
- Полностью API — ~$15 500

## Чек-лист перед началом сборки нового поколения

Перед запуском процесса убедись:

- [ ] Промпты 1-7 актуальны и под рукой
- [ ] Знаешь точное название поколения и годы выпуска
- [ ] Знаешь все двигатели и коробки этого поколения
- [ ] Не путаешь поколение с предыдущим/следующим (например, Solaris RB 2010-2017 vs HC 2017-2020)
- [ ] Готов потратить 7-9 часов вручную (с защитами 5.3 и 7.1)
- [ ] Понимаешь что **шаги 5.3 и 7.1 нельзя пропускать** — это защита от галлюцинаций

## Чек-лист контроля качества при сборке

Во время каждого шага проверяй:

**Шаг 5.3 (web-search верификация):**
- [ ] Все `recall_id` подтверждены в первоисточнике (NHTSA / Росстандарт)
- [ ] Все `case_number` найдены в правовых базах
- [ ] Все `part_number` для systemic_defect найдены в OEM-каталогах
- [ ] Конфликты Claude vs Grok разрешены через первоисточник
- [ ] Неподтверждённые данные → `null + needs_review: true`

**Шаг 7.1 (проверка источников аудитора):**
- [ ] Каждая «пропущенная болячка» подтверждена открытыми источниками
- [ ] Каждый «правильный артикул» подтверждён в каталоге OEM
- [ ] Каждый «пропущенный recall» найден в NHTSA / Росстандарт
- [ ] Неподтверждённые правки аудитора → пропущены, отмечены в MERGE_REPORT

## Главный урок

**Без шага 6 (аудит) в БД попадают опасные ошибки.** На Solaris RB это был ATF SP-III/SP-IV — артикул, который пользователь увидел бы в приложении и залил в коробку, после чего она бы вышла из строя.

**Аудит обязателен. Без исключений.**

## Две защиты от галлюцинаций (обновление v3.0)

В версии 3.0 процесса добавлены два новых подшага защиты:

### Защита №1: Шаг 5.3 — web-search верификация ПЕРЕД сборкой

**Проблема которую решает:** Claude и Grok на шагах 1-2 могут выдумать артикул или recall_id. Если просто слить их данные — галлюцинация попадает в JSON.

**Как защищает:** перед записью в JSON каждый артикул и recall_id проверяется в первоисточнике (NHTSA, OEM-каталоги). Если не подтверждается — `null + needs_review: true`.

### Защита №2: Шаг 7.1 — повторная проверка источников аудитора ПОСЛЕ аудита

**Проблема которую решает:** аудитор тоже LLM, тоже может галлюцинировать. Может «найти» болячку которой нет, или предложить «правильный» артикул который тоже неправильный.

**Как защищает:** перед применением каждой правки аудитора — проверка её источников через web-search. Не подтверждается — правка не применяется.

### Стоимость защиты

Эти два подшага замедляют процесс на ~30 минут на каждое поколение. Но они **исключают опасные галлюцинации** в production. Стоимость замедления = ~20% времени работы. Стоимость пропущенной галлюцинации = поломка пользователя на $2000 + репутационный риск приложения.

Решение: всегда выполнять 5.3 и 7.1.
